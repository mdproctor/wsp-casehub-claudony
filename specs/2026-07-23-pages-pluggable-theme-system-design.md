# Pages Pluggable Theme System

**Target repo:** casehubio/pages (pages-ui-tokens package)
**Date:** 2026-07-23
**Status:** Draft

## Summary

Replace pages-ui-tokens' monolithic runtime theme generation with a build-time pipeline architecture. Themes are composable pipelines of transforms declared in JSON presets. The pipeline generates DTCG-standard token files + CSS. The browser runtime becomes thin — apply a pre-built theme by name, no generation logic.

## Motivation

The current system has four limitations:

1. **No named themes** — one `ThemeConfig` object, two modes (light/dark). No way to express "CaseHub brand" vs "pages generic" vs "customer X brand."
2. **Runtime generation waste** — every app runs `generateThemeCSS()` in the browser to produce identical output. Theme values are design decisions, not runtime computations.
3. **Flat generation** — 4 parameters (baseHue, accentHue, chroma, contrast) with hardcoded semantic hues and linear chroma. Can't reproduce the CaseHub brand palette (achromatic neutrals, teal success, shifted dark lightness range).
4. **No semantic tier** — components use primitive step numbers (`neutral-2`, `neutral-3`) directly. No explicit `surface-primary`, `text-muted` roles. Theme switching is fragile.

## Architecture

### Three-tier token hierarchy

```
Tier 1: Primitives (generated by pipeline)
  --pages-neutral-1 .. --pages-neutral-12
  --pages-accent-1  .. --pages-accent-12
  --pages-success-1 .. --pages-success-12
  --pages-violet-1  .. --pages-violet-12   (brand hue — casehub-dark)
  (12-step OKLCH scales — names come from the oklch-scale hues param)

Tier 2: Role tokens (mapped by pipeline via semantic-map)
  --pages-surface-primary    → neutral-1
  --pages-surface-secondary  → neutral-2
  --pages-surface-tertiary   → neutral-3  (table stripes, nested panels)
  --pages-border-subtle      → neutral-4
  --pages-border-default     → neutral-6
  --pages-text-primary       → neutral-12
  --pages-text-secondary     → neutral-11
  --pages-text-muted         → neutral-8
  --pages-interactive        → accent-9
  --pages-interactive-hover  → accent-10
  --pages-focus-ring         → accent-8
  (stable abstraction for theme-portable components)

Tier 3: Component (app-level, optional — not in the theme system)
  Apps may alias role or primitive tokens for convenience, but this
  is app-level CSS, not theme-system scope.
```

Scale names in tier 1 are preset-defined via the `oklch-scale` transform's `hues` parameter. The current implementation uses names matching semantic categories (`neutral`, `accent`, `success`, `warning`, `danger`, `info`). Brand presets add colour-named scales (`violet`, `green`, `magenta`). There is no enforced naming convention for scale names — the `semantic-map` transform maps whatever scales exist to role tokens.

### Pipeline architecture

A theme is an ordered list of composable transforms. Each transform is a pure function: `TokenMap → TokenMap`.

**TokenMap** is a recursively nested DTCG-structured object. Leaf nodes carry `$value` and `$type`; intermediate nodes are grouping containers:

```typescript
type TokenLeaf = { $value: string; $type: string };
type TokenMap = { [key: string]: TokenLeaf | TokenMap };
```

Transforms receive the full accumulated TokenMap from prior transforms and return a new TokenMap. This means transforms **compound** — a `chroma-curve` running after `oklch-scale` operates on the scale's output, not on empty state. This is defined semantics, not a side effect.

The pipeline array is sequential — transforms execute in declaration order. Two categories exist as a **naming convention** for preset authors, not as enforcement:
- **Algorithmic** — deterministic transforms from parameters. Conventionally listed first.
- **Manual** — explicit token overrides. Conventionally listed last so they win.

A preset is a JSON file declaring its pipeline:

```json
{
  "$schema": "https://pages.casehub.org/preset-schema.json",
  "$name": "casehub-dark",
  "$description": "CaseHub brand dark theme",
  "$extends": "default-dark",
  "pipeline": [
    { "transform": "oklch-scale", "params": { "hues": { "violet": 270, "green": 160, "magenta": 320 } }},
    { "transform": "lightness-shift", "params": { "offset": 10 }},
    { "transform": "chroma-curve", "params": { "curve": "gaussian", "neutral": 0.02 }},
    { "transform": "semantic-hues", "params": { "success": 175, "warning": 100 }}
  ]
}
```

Note that `oklch-scale` here adds three brand hue scales on top of the scales already generated by `default-dark`'s pipeline. Because `oklch-scale` is additive and transforms compound via `$extends`, the child's brand scales merge alongside the parent's base scales.

`$extends` runs the parent's full pipeline first, then appends the child's transforms. This is **append-only by design** — a child cannot remove or replace a parent transform. If a child needs fundamentally different pipeline structure, it should not extend the parent.

**`$extends` resolution:** Built-in preset names (`default-light`, `default-dark`, `casehub-dark`, etc.) resolve from an internal registry bundled with pages-ui-tokens. Custom presets can extend built-ins by name or other custom presets by relative file path (e.g. `"$extends": "./base-brand.json"`). Circular extends and unresolvable names are build errors with explicit diagnostics.

### Built-in transforms

**Algorithmic:**

| Transform | Purpose | Key params |
|-----------|---------|------------|
| `oklch-scale` | Generate 12-step OKLCH primitive scales per hue | `hues: Record<string, number>`, `chroma`, `contrast` |
| `light-mode` / `dark-mode` | Select lightness step array | — |
| `lightness-shift` | Uniform offset to all steps | `offset` (positive = brighter) |
| `lightness-steps` | Replace the 12-step array entirely | `steps: number[]` |
| `chroma-curve` | Control chroma distribution across the scale | `curve: 'flat' \| 'gaussian' \| 'bezier'`, per-semantic multipliers (e.g. `neutral: 0.02`) |
| `semantic-hues` | Override default semantic hue angles | `success`, `warning`, `danger`, `info` |
| `semantic-map` | Map primitive scales to semantic roles + generate role tokens | `neutral: 'gray-cool'`, `accent: 'blue'`, role mappings |
| `contrast-check` | Validate APCA contrast ratios | `minContrast`, `fix: boolean` |
| `gamut-clamp` | Clamp out-of-gamut OKLCH values to sRGB | — |

**Manual:**

| Transform | Purpose | Key params |
|-----------|---------|------------|
| `override` | Set specific tokens to exact values | Map of token name → value |

**Transform behavioral details:**

- **`oklch-scale`** is additive — each invocation merges new hue scales into the TokenMap alongside existing ones. The `hues` parameter maps scale names to hue angles (e.g. `{ "blue": 240, "teal": 175, "violet": 270 }`). This replaces the old `baseHue`/`accentHue` pair — any number of named hue scales can be generated.
- **`contrast-check`** with `fix: false` (the default) fails the build with diagnostic output listing every pair that violates the `minContrast` APCA threshold. With `fix: true`, it adjusts lightness steps to meet the threshold, clamps chroma to keep the result in gamut, and logs every adjustment as a warning. CI pipelines should use `fix: false`; the standalone design tool uses `fix: true` for interactive exploration.
- **`gamut-clamp`** uses CSS Color Level 4 binary-search chroma reduction: for each out-of-gamut OKLCH value, reduce chroma toward zero while preserving lightness and hue until the value falls within sRGB. This is the same algorithm browsers use for gamut mapping and produces consistent results with CSS `oklch()` rendering.

### Built-in presets

| Preset | What it is | Extends |
|--------|-----------|---------|
| `default-light` | Pages generic light | — (base) |
| `default-dark` | Pages generic dark | — (base) |
| `casehub-light` | CaseHub brand light | `default-light` |
| `casehub-dark` | CaseHub brand dark (Claudony / casehub.org look) | `default-dark` |

The `casehub-dark` preset targets the current Claudony/casehub.org palette:
- Near-achromatic neutrals (chroma ~0.002-0.006 vs default's ~0.018)
- Dark range shifted brighter (background at L=17.8 vs default's L=8)
- Accent hue 240 (CaseHub blue) at higher chroma (~0.145)
- Brand hue scales for the "Bioluminescent Colony" identity: violet (270), green (160), magenta (320)
- Success hue 175 (teal), warning hue 100 (warm yellow-green)

Whether the algorithmic transforms can reproduce the exact CaseHub look will be validated during implementation. If gaps remain, `override` transforms fill them. The pipeline debug command (`pages-tokens debug`) dumps token state after each step to identify where the generation diverges from the target.

The `casehub-light` preset applies the same brand identity (hues, chroma curves) adapted for light mode. Its exact values will be designed using `pages-tokens debug` to iterate on parameters once the pipeline is operational.

## Preset format (DTCG-aligned)

Preset JSON files follow DTCG v2025.10 conventions:
- `$`-prefixed reserved properties (`$name`, `$description`, `$schema`)
- `$extends` for preset inheritance (pages extension, not DTCG standard)
- `pipeline` array is the pages-specific content

Pipeline output is DTCG-standard token JSON:

```json
{
  "$name": "casehub-dark",
  "color": {
    "surface": {
      "primary": { "$value": "oklch(17.8% 0.002 260)", "$type": "color" },
      "secondary": { "$value": "oklch(20.6% 0.003 260)", "$type": "color" }
    }
  }
}
```

Plus generated CSS custom properties for direct consumption.

## Build pipeline and CLI

The pipeline runner is a Node library + CLI. Same code path for both.

**CLI:**

```bash
pages-tokens build                              # all presets in presets/
pages-tokens build presets/casehub-dark.json     # specific preset
pages-tokens build ./my-brand.json --out dist/   # custom preset
pages-tokens validate presets/*.json             # validate without building
pages-tokens debug presets/casehub-dark.json     # dump state after each step
```

**Output per preset:**

```
dist/
├── casehub-dark.css              ← CSS custom properties
├── casehub-dark.tokens.json      ← DTCG token JSON
```

**Build integration:** `npm run build` in pages-ui-tokens runs `pages-tokens build` before `tsc`. CI runs `pages-tokens validate` to catch broken presets.

**Standalone tool:** Same pipeline with a browser UI. User adjusts parameters, previews results, exports a JSON preset file to check into their project. The tool itself uses the pipeline library — no separate generation logic.

### CSS output structure

Each preset generates CSS custom properties scoped under a `.pages-theme-{$name}` class selector:

```css
.pages-theme-casehub-dark {
  --pages-neutral-1: oklch(17.8% 0.002 260);
  --pages-neutral-2: oklch(20.6% 0.003 260);
  /* ... all primitive and semantic tokens */
  --pages-surface-primary: var(--pages-neutral-1);
  --pages-text-primary: var(--pages-neutral-12);
}
```

`applyTheme('casehub-dark')` sets `class="pages-theme-casehub-dark"` on `document.documentElement` (or a specific target element when provided). SSR emits `<html class="pages-theme-casehub-dark">` directly. Consumers can scope overrides under the same selector. Multiple themes can coexist in the DOM via class scoping on different subtrees — `applyTheme('brand-x', panelEl)` scopes one theme to a subtree while the document root uses another.

## Runtime API

The runtime is thin — no pipeline logic, no generation.

```typescript
// Apply a built-in or registered theme (document-level)
applyTheme('casehub-dark');

// Apply to a specific subtree element (e.g. Pages site embedded in a panel)
applyTheme('casehub-dark', panelElement);

// Register a custom theme (pre-built CSS string)
registerTheme('brand-acme', acmeCssString);
applyTheme('brand-acme');

// Query
getTheme(target?: HTMLElement): string;
listThemes(): string[];
```

`applyTheme(name: string, target?: HTMLElement)` injects a `<style data-pages-theme>` element and sets the theme class on `target` (defaults to `document.documentElement`). The `target` parameter preserves element-scoped theming — `pages-runtime` uses this to theme a `loadSite` subtree independently of the document root.

**Obtaining CSS for `registerTheme`:**

Built-in themes are auto-registered at import time — their CSS is bundled with pages-ui-tokens. For custom themes, three paths:

1. **Build-time import** — import the generated CSS file as a string via bundler raw import (e.g. `import css from './dist/brand-acme.css?raw'`). Most common for apps that own their preset.
2. **Dynamic fetch** — `fetch('/themes/brand-acme.css').then(r => r.text()).then(css => registerTheme('brand-acme', css))`. For multi-tenant or plugin scenarios where themes are loaded on demand.
3. **Inline in JS** — the build pipeline can optionally emit a `.js` module that exports the CSS string and self-registers. Convenience wrapper, not a separate generation path.

### What changes from today

| Today | After |
|-------|-------|
| `injectTheme(DEFAULT_THEME)` — runs generation in browser | `applyTheme('default-dark')` — applies pre-built CSS |
| `applyThemeMode(el, 'dark')` — toggles mode on one config | `applyTheme('casehub-dark', el)` — switches to any named theme, optionally scoped |
| `generateThemeCSS(config)` — browser export | Build-only export — removed from runtime |
| `ThemeConfig` object at runtime | Theme name string at runtime |

## Semantic role tokens

Components use role tokens instead of primitive step numbers. This ensures consistent appearance across all components in any theme.

**Surface roles:**

| Token | Purpose | Typical mapping |
|-------|---------|----------------|
| `--pages-surface-primary` | Page background | neutral-1 |
| `--pages-surface-secondary` | Card/panel background | neutral-2 |
| `--pages-surface-tertiary` | Nested surface, table stripe | neutral-3 |
| `--pages-surface-hover` | Hover state overlay | neutral-3 (light) / neutral-4 (dark) |
| `--pages-surface-selected` | Selected row/item | accent-2 |

**Border roles:**

| Token | Purpose | Typical mapping |
|-------|---------|----------------|
| `--pages-border-subtle` | Dividers, separators | neutral-4 |
| `--pages-border-default` | Input borders, card borders | neutral-6 |
| `--pages-border-strong` | Focused input, emphasis | neutral-8 |

**Text roles:**

| Token | Purpose | Typical mapping |
|-------|---------|----------------|
| `--pages-text-primary` | Headings, emphasis | neutral-12 |
| `--pages-text-secondary` | Body text | neutral-11 |
| `--pages-text-muted` | Labels, captions, timestamps | neutral-8 |
| `--pages-text-disabled` | Disabled controls | neutral-6 |

**Interactive roles:**

| Token | Purpose | Typical mapping |
|-------|---------|----------------|
| `--pages-interactive` | Buttons, links | accent-9 |
| `--pages-interactive-hover` | Hover state | accent-10 |
| `--pages-interactive-active` | Active/pressed state | accent-11 |
| `--pages-focus-ring` | Focus indicator | accent-8 |

**Status roles:**

| Token | Purpose | Typical mapping |
|-------|---------|----------------|
| `--pages-status-success` | Success states | success-9 |
| `--pages-status-warning` | Warning states | warning-9 |
| `--pages-status-danger` | Error/danger states | danger-9 |
| `--pages-status-info` | Informational states | info-9 |

The `semantic-map` transform generates these role tokens from the primitive scales. Different presets can map to different steps — `casehub-dark` might map `surface-primary` to `neutral-2` (brighter background) while `default-dark` maps it to `neutral-1`.

**Brand hue consumption:** Brand-specific hue scales (e.g. `--pages-violet-*`, `--pages-green-*`, `--pages-magenta-*` from the casehub-dark preset) are consumed as tier 1 primitives. They sit outside the semantic role abstraction by design — brand colours are identity-specific and not intended to be theme-portable. Components that use brand hues directly are brand-specific components (e.g. Claudony's landing page bioluminescent effects).

For presets that need custom role tokens beyond the predefined set, the `semantic-map` transform's role mappings are extensible. A preset can define additional mappings like `brand-primary → violet-9`, `brand-glow → green-7` to generate custom `--pages-brand-primary`, `--pages-brand-glow` role tokens. These custom roles are preset-specific — they exist only when the preset defines them.

## Non-color tokens

Spacing, typography, motion, and radius tokens are static constants — they do not vary per theme. They are included in every preset's DTCG output and CSS for completeness but are not pipeline-parameterised. The `semantic-map` transform references them when generating composite tokens (e.g. a focus ring that combines an accent colour with a radius value).

**Density variants** remain a separate mechanism: the existing `.pages-density-compact` class applies `DENSITY_COMPACT_OVERRIDES` independently of theme selection. Density is orthogonal to colour theming.

**Elevation tokens** (shadow definitions) differ between light and dark modes. The `light-mode`/`dark-mode` transforms select the appropriate elevation set (`ELEVATION_LIGHT` or `ELEVATION_DARK` from `tokens.ts`), and the pipeline includes them in the output alongside colour tokens.

## Coherence with existing theme APIs

### Token name reconciliation

Three sets of CSS custom property names exist in the ecosystem, at different architectural tiers:

1. **Legacy pages-ui names** (`--pages-text`, `--pages-bg`, `--pages-accent`, `--pages-border`) — these predate pages-ui-tokens entirely. They are documented in CASEHUB-PAGES.md but the corresponding `PagesTheme` type and `LIGHT_THEME`/`DARK_THEME` constants no longer exist in code. These names are deprecated.
2. **pages-ui-tokens primitives** (`--pages-neutral-1` through `--pages-neutral-12`, `--pages-accent-*`, etc.) — tier 1, generated by the pipeline's `oklch-scale` transform. These continue to exist and are stable.
3. **Semantic role tokens** (`--pages-surface-primary`, `--pages-text-muted`, `--pages-interactive`, etc.) — tier 2, new in this spec. Generated by the `semantic-map` transform. These are the recommended consumption point for components.

App-level aliases (e.g. claudony's `--bg: var(--pages-neutral-1)`) are tier 3 — outside the theme system's scope. Apps may continue using their own aliases or migrate to semantic role tokens.

### LiveSite.setTheme evolution

`LiveSite.setTheme(mode: 'light' | 'dark')` currently calls `applyThemeMode(target, mode)` internally, where `target` is the element passed to `loadSite()`. After this spec, it evolves to call `applyTheme(resolvedName, target)` — preserving element-scoped theming. The `setTheme` signature remains stable — it maps mode strings to theme names via a configurable lookup. See §Migration > pages-runtime for the full migration path.

The old runtime generation functions (`generateThemeCSS`, `injectTheme`) become build-only exports — they are not removed from the package, but they are removed from the runtime API surface. Import paths change: `import { generateThemeCSS } from 'pages-ui-tokens/build'` instead of the top-level export.

## Migration

### pages-ui-tokens (breaking — 0.3.0)

- Add `src/transforms/` — the transform library
- Add `src/pipeline.ts` — the pipeline runner
- Add `src/cli.ts` — the CLI wrapper
- Add `presets/` — built-in preset JSON files
- Add `dist/` output generation to build
- Modify `src/themes.ts` — move `generateThemeCSS`, `injectTheme` to build-only exports
- Modify `src/index.ts` — export `applyTheme`, `registerTheme`, `listThemes`, `getTheme`; remove runtime generation exports
- Keep `src/colours.ts` — `generateScale` becomes internal to the `oklch-scale` transform
- Keep `src/tokens.ts` — spacing, typography, motion, radius constants used by `semantic-map` transform

### pages-runtime

pages-runtime has deeper integration than standalone consumer apps:

1. `import type {ThemeConfig}` and `SiteOptions.themeConfig?: ThemeConfig` — `ThemeConfig` is removed from the runtime API surface. `SiteOptions` replaces `themeConfig` with `themeName?: string` (e.g. `'casehub-dark'`).
2. `applyThemeMode(target, mode)` in `loadSite` — changes to `applyTheme(themeName, target)`, using the element-scoped variant to preserve subtree theming.
3. `LiveSite.setTheme(mode)` — internally calls `applyTheme(resolvedName, target)` instead of `applyThemeMode(target, mode)`. The mode-to-name resolution uses a configurable lookup (defaulting to `{ light: 'default-light', dark: 'default-dark' }`).
4. `injectTheme` import — removed; theme CSS is pre-built and injected by `applyTheme`.

### Consumer apps (one-line change each)

Standalone consumer apps (not pages-runtime) replace:
```typescript
injectTheme(DEFAULT_THEME);
applyThemeMode(document.documentElement, 'dark');
```
with:
```typescript
applyTheme('casehub-dark');
```

Affected: claudony, chat-app, blocks-ui examples.

### blocks-ui components

Components currently using `var(--pages-neutral-N)` directly should migrate to semantic role tokens (`var(--pages-surface-secondary)`, `var(--pages-text-muted)`, etc.) over time. This is not blocking — the primitive tokens still exist and still work. The semantic tier is additive.

**Test infrastructure update:** The `INVALID_PAGES_VARS` regex in `theme-variables.test.ts` (channel-activity component) currently rejects old-style names like `--pages-bg` and `--pages-text` while allowing primitive tokens. This regex must be updated to also accept the new semantic role token names (`--pages-surface-*`, `--pages-text-*`, `--pages-border-*`, `--pages-interactive-*`, `--pages-status-*`, `--pages-focus-*`). Without this update, the test will block semantic token adoption in blocks-ui components.

## Testing

- **Transform unit tests** — each transform tested in isolation with known input/output TokenMaps
- **Pipeline integration tests** — full pipeline runs on built-in presets, asserts output structure and DTCG compliance
- **Contrast validation** — `contrast-check` transform runs in CI on all presets, fails on APCA violations
- **Gamut validation** — `gamut-clamp` ensures all colours are sRGB-representable
- **Snapshot tests** — generated CSS for each preset is snapshot-tested to catch unintended drift
- **Visual regression** — blocks-ui showcase rendered with each preset, compared against baseline screenshots
- **Consumer smoke tests** — Claudony and chat-app build with the new runtime API

## Execution order

1. Transform library + pipeline runner + CLI
2. Migrate `default-light` and `default-dark` as pipeline presets (must produce identical output to current `generateThemeCSS`)
3. Design and tune `casehub-dark` preset (match Claudony/casehub.org palette)
4. Design `casehub-light` preset
5. Runtime API (`applyTheme`, `registerTheme`)
6. Migrate consumers (claudony, chat-app, blocks-ui examples)
7. Semantic role token migration in blocks-ui components (can be incremental)
8. Standalone designer tool (follow-on)

## Out of scope

- Standalone designer tool UI (follow-on work — the pipeline library is the foundation)
- Component-level token overrides (tier 3 — app responsibility, not theme system)
- Nested theme resolution (automatic inheritance from parent theme scope)
- CSS-in-JS or styled-components integration
- Non-web targets (iOS, Android token generation via Style Dictionary)
